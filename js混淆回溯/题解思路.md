# 猿人学第6题 - JS混淆回溯 解题思路

## 题目现象

- 爬虫请求第 1 页成功，第 2~5 页返回 `风控不通过，别匆忙，放慢速度`
- 同样的加密参数，第 1 页能过，后面的过不了
- 浏览器中正常点击 5 页都通过，但调试时停顿太久反而会触发风控

## 排错过程

### 1. 猜测：鼠标监听注入的熵（Ae）

JS 中有 `window.addEventListener("mousemove", Be)`，怀疑 Node.js 没有鼠标事件导致 Ae 数组为空，后续页面的 RSA 密钥没有随机性。

**结论：❌ 排除**
- 插桩发现 `Be` 函数体内的条件 `this.count >= 256 || Ee >= Me` 在首次调用时永远为 `false`，**body 永远不会执行**
- Ae 在每次调用时都是全 0，且 clean 版本和当前版本一致

### 2. 猜测：AA-encode 头部被改

当前文件开头的颜文字混淆与 clean 版本不同，怀疑影响了初始化逻辑。

**结论：⚠️ 不是主因**
- 替换为 clean 版本的头部后 ASN1 引用报错，修复后运行正常
- 但页 2~5 依然风控，说明问题不在这里

### 3. 猜测：请求头差异

浏览器带了 `Referer`、`X-Requested-With`、完整 UA，Python 只带了 `yuanrenxue`。

**结论：❌ 排除**
- 第 1 页没这些头也能过
- 补上这些头后页 2~5 依然风控

### 4. 真相：模块缓存 `_n` 与闭包 `Re`

**核心机制：**

```javascript
// 模块加载器 — 缓存机制
function n(t) {
    if (s[t]) return s[t].exports;  // 已加载过 → 返回缓存
    var e = s[t] = { ... };
    return i[t].call(e.exports, e, e.exports, n), e.exports;
}
_n = n;

// IIFE 里的闭包单例 — Re 是跨调用共享的
(function() {
    var Re, Ae, Ee;       // ← 闭包变量，模块缓存一次就永久存活

    function ne() {
        if (null == Re) {        // 只初始化一次
            Re = se();
            // 用 Ae 填充 Re (RC4)
        }
        return Re.next();        // 每次调用推进 RC4 流
    }

    function oe() {}
    oe.prototype.nextBytes = re; // re 调 ne()
    // ...
    // 所有导出的原型方法都引用这个闭包
    // → 闭包因 _n 的缓存而永不释放
    // → Re 的状态一直持续到进程结束
})();
```

**为什么浏览器能过：**
```
浏览器同一页面（一个进程）：
  第 1 次 get_value → ne() → Re 首次初始化 → 消耗 RC4 字节 0~N   → 密钥 A
  第 2 次 get_value → ne() → Re 已存在     → 消耗 RC4 字节 N+1~M → 密钥 B（不同！）
  第 3 次 get_value → ne() → Re 已存在     → 消耗 RC4 字节 M+1~K → 密钥 C（不同！）
  ...
```

**为什么爬虫不过（每页新进程）：**
```
进程1（第 1 页）→ 全新 Re → 字节 0~N → 密钥 A
进程2（第 2 页）→ 全新 Re → 字节 0~N → 密钥 A（和进程1完全一样！）
进程3（第 3 页）→ 全新 Re → 字节 0~N → 密钥 A（又一样！）
```

服务器看到第 2 页和第 1 页的 RSA 密钥相同 → 检测为非浏览器行为 → 风控。

## 解决方案

**把 5 页请求放在同一个 Node.js 进程里执行**，让 RC4 流连续推进：

```javascript
// JS 端 — 一次性生成 5 页的加密参数
var base = Date.now();
var results = [];
for (var p = 1; p <= 5; p++) {
    results.push(JSON.parse(get_value(base + p)));
}
console.log(JSON.stringify(results));
```

```python
# Python 端 — 只调用一次 Node.js
def call_js_all():
    proc = subprocess.run(['node', '颜文字混淆解码.js'],
        capture_output=True, text=True, encoding='utf-8', check=True)
    return json.loads(proc.stdout)

all_data = call_js_all()  # 一次拿到 5 页的加密参数
for page, data in enumerate(all_data, 1):
    # 正常发请求...
```

## 关键知识点

| 知识点 | 说明 |
|-------|------|
| **模块缓存 `_n`** | webpack 风格的模块加载器，`_n("encrypt")` 首次执行 IIFE，之后返回缓存 |
| **闭包生命周期** | IIFE 的闭包被导出函数引用 → `_n` 缓存了导出 → 闭包**永不释放** |
| **`Re` 是单例** | `ne()` 里的 `Re` 是闭包变量，不是实例属性，所有 `new oe()` 共享同一个 RC4 状态 |
| **RC4 流式推进** | 每次 `Re.next()` 返回不同字节，连续调用产生不同的随机数序列 |
| **独立进程问题** | `subprocess.run` 每次新进程 → 新闭包 → Re 重新初始化 → 序列重复 |

## 教训

- **不要只看单次调用内部**：加密函数每条路径看起来都独立，但共享的闭包状态在"调用之间"
- **模块缓存的副作用**：`_n` 的 `s` 对象缓存了整个模块，包括 IIFE 的闭包
- **插桩要跨调用**：不仅要看单次加密的输入输出，还要看跨调用的状态变量（`Re`、`Ee` 等）
